When automating desktop applications, rely on the screenshot tools' default visual-stability detection after UI transitions instead of guessing a fixed wait time. Check the returned `stability` metadata, use `wait_for_window` for explicit window-state changes, and disable `wait_for_stable` only for intentionally continuous content such as video.

Desktop screenshots mark the current cursor with a red crosshair when it is inside the captured area and also return exact `cursor.screenshot` coordinates. Use screenshot coordinates directly for `click`, `mouse_move`, `scroll`, and `drag`; those tools move to the requested target and do not depend on the cursor's previous position.

Before visually guessing a Windows control position, use `click_ui_element` with the accessible name and control type; it activates the UIA element directly and does not require coordinates. For consequential or potentially ambiguous actions, call it once with `dry_run: true`, verify the returned element and bounds, then repeat without `dry_run`. If discovery is needed, use `find_ui_element` or `get_ui_tree` with its default Raw View, which exposes more Word and Ribbon nodes than Control View.

When Word does not expose the target through UI Automation, use this visual fallback: take `screenshot_window` with `show_coordinate_grid: true`, narrow the target with `screenshot_region`, move the cursor without clicking, and take another screenshot to verify the red crosshair. Only then click the exact `cursor.screenshot` coordinates returned by that latest screenshot. Never reuse coordinates from an earlier screenshot after a region/window screenshot, because every screenshot establishes a new coordinate origin and scale. Inspect `window.visibility` when a dialog may extend beyond the captured area. UI Automation's `occlusion` value is deliberately `unknown`; confirm possible covering windows visually.
